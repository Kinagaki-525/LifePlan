# UI実装ガイドライン

## 1. 目的

このドキュメントは、Figmaで作成したUIデザインを、このリポジトリのWebアプリへ実装する際の共通方針を定める。

Figmaは見た目と体験の基準とし、実装では既存のASP.NET Core MVC、Razor、Bootstrap、CSSの構成に合わせて変換する。

## 2. 基本方針

- FigmaをUIの正とする。
- 実装は既存の技術スタックに合わせる。
- Figmaから取得したReact、Tailwind、絶対配置コードは、そのまま貼り付けない。
- 入力仕様、ViewModel、Validator、Mapper、Domainの契約は、UI調整だけを目的として変更しない。
- UI変更は、見た目、余白、配置、ラベル、補足文、エラー表示、レスポンシブ表示を中心に行う。
- 大きな画面構成変更や入力仕様変更が必要な場合は、UI実装とは別の仕様変更として扱う。
- Figmaの見た目と既存の入力仕様、アクセシビリティ、POST仕様が衝突する場合は、既存の動作仕様を優先し、その範囲でFigmaに近づける。

## 3. 対象技術への変換ルール

Figma連携ツールから生成される参照コードは、React + Tailwind形式になる場合がある。このリポジトリでは、以下のように変換する。

| Figma / 生成コード | このリポジトリでの扱い |
| --- | --- |
| Reactコンポーネント | Razorビュー、Partial View、HTML構造へ読み替える |
| Tailwindクラス | Bootstrapクラスまたは `wwwroot/css/site.css` のCSSへ変換する |
| 絶対配置 | 通常の文書フロー、Bootstrap Grid、Flexbox、CSS Gridへ変換する |
| Figmaコンポーネント | 再利用できるHTML構造またはCSSクラスとして整理する |
| Figmaカラー変数 | CSSカスタムプロパティまたは共通CSSクラスへ反映する |
| Figmaテキストスタイル | 見出し、本文、補足文のCSSルールへ反映する |
| Figma画像アセット | `wwwroot` 配下の静的ファイルとして管理する |

Tailwindは導入しない。既存プロジェクトのCSS方針を優先する。

## 4. デザイントークンの扱い

Figmaで定義されている色、フォント、余白、角丸、影は、可能な範囲でCSSに集約する。

パディング、マージン、フォントサイズ、角丸、影、主要カラーは、画面ごとに個別指定を増やさず、プロジェクト共通の定数として扱う。CSSでは、CSSカスタムプロパティまたは共通ユーティリティクラスとして定義し、アプリ全体で統一感を保つ。

デザイントークンは、原則として `wwwroot/css/site.css` の `:root` に定義する。画面固有CSSが必要な場合でも、色、余白、フォントサイズなどの値は共通トークンを参照する。

例:

```css
:root {
  --ui-color-primary: #4caf7d;
  --ui-color-background: #fff8f0;
  --ui-color-text: #333333;
  --ui-color-muted: #888888;
  --ui-color-border: #e0e0e0;
  --ui-font-size-body: 1rem;
  --ui-font-size-small: 0.875rem;
  --ui-font-size-heading: 2rem;
  --ui-space-sm: 0.5rem;
  --ui-space-md: 1rem;
  --ui-space-lg: 1.5rem;
  --ui-radius-md: 0.5rem;
}
```

ただし、既存CSSに同等の変数やクラスがある場合は、既存定義を優先する。

新しい値を追加する前に、既存のCSS変数、Bootstrapのspacing class、共通CSSクラスで代替できないか確認する。

新規トークンは、既存スケールで表現できない場合のみ追加する。Figma上のpx値をそのまま増やし続けず、近い値がある場合は既存トークンへ寄せる。px直書きは、Figma再現に必要で共通化しづらい例外に限定する。

## 5. レイアウト変換ルール

- Figmaのフレームは、Razor上では `section`、`div`、フォームグループなどの意味のある構造へ変換する。
- 画面全体の背景、ヘッダー、フッター、メイン領域は、既存の `_Layout.cshtml` と整合させる。
- PC表示で横並びの項目は、Bootstrap GridまたはFlexboxで実装する。
- スマホ表示では1カラムを基本とし、入力欄が横にはみ出さないようにする。
- Figma上のpx値は参考値とし、実装では保守しやすい余白スケールへ丸めてよい。
- 画面幅に依存する固定配置は避ける。

## 6. フォーム実装ルール

- `asp-for` は既存のViewModel契約を維持する。
- POSTされるフィールド名をUI都合で変更しない。
- ラベルには、入力値の意味と単位が分かる文言を使う。
- 補足文は、入力判断に必要なものだけを短く表示する。
- `asp-validation-for` は対象入力項目の近くに配置する。
- バリデーションサマリーは、画面全体のエラー把握用として残す。
- 条件付き入力は、入力不要な状態と入力可能な状態が視覚的に分かるようにする。
- disabledにするとPOSTされないため、送信が必要な値には安易に使わない。

## 7. コンポーネント変換ルール

- Figmaのボタン、タブ、カード、入力ブロックは、共通CSSクラスとして再利用できる形に整理する。
- Bootstrapで表現できるものはBootstrapを優先する。
- Bootstrapだけで表現しづらいブランド色、角丸、影、タブ表現などは `site.css` に追加する。
- 同じ見た目を複数箇所で使う場合は、都度インラインで書かず、共通クラス化する。
- カードの中に不要なカードを重ねない。

## 8. 重複チェック

UI実装時は、HTML構造とCSSの重複を確認し、同じ意図の実装が増えすぎないようにする。

- 同じ見た目のボタン、タブ、カード、入力ブロックが複数ある場合は、共通クラス化を検討する。
- 同じ `padding`、`margin`、`font-size`、`border-radius`、`box-shadow` を複数箇所に直接書かない。
- 既存CSSに近いクラスがある場合は、新しいクラスを追加する前に再利用または拡張を検討する。
- Razor内で同じHTML構造が繰り返される場合は、Partial View、Tag Helper、ループ化、共通CSSクラスのいずれかで整理できないか確認する。
- ただし、抽象化によってRazorの可読性が下がる場合は、無理に共通化しない。
- UI部品の意味が異なる場合は、見た目が近くても別クラスとして扱ってよい。

## 9. アクセシビリティ

- 入力項目には必ず対応するラベルを置く。
- 見出し階層は `h1`、`h2`、`h3` の順序を崩さない。
- ボタンとリンクは役割に応じて使い分ける。
- 色だけで状態を伝えない。
- エラー文はスクリーンリーダーでも意味が伝わる文にする。
- キーボード操作でフォーム入力と送信ができる状態を維持する。
- タブ、ツールチップ、ポップオーバー、開閉パネルなど状態を持つUIは、見た目だけでなくARIA構造も確認する。
- タブUIでは `role="tablist"`、`role="tab"`、`role="tabpanel"`、`aria-controls`、`aria-labelledby`、`aria-selected` の整合を確認する。
- ツールチップや参考情報は、装飾ではない情報を `aria-hidden` にしない。

## 10. 実装手順

1. Figmaの対象フレーム、PC表示、スマホ表示を確認する。
2. 既存のRazorビュー、ViewModel、CSS、レイアウトを確認する。
3. 既存のCSSクラス、Bootstrap利用箇所、Razorの繰り返し構造を確認し、再利用できるUIパターンを把握する。
4. Figmaの構造を、このリポジトリのRazor構造へ対応付ける。
5. 必要なデザイントークンをCSSへ追加または既存CSSへ統合する。
6. Razorでは `asp-for`、`asp-validation-for`、既存のフォーム送信構造を維持して見た目を調整する。
7. CSSやHTMLに重複が増えていないか確認する。
8. PC幅とスマホ幅で表示を確認する。
9. 初期表示、POST、バリデーションエラー、入力値保持を確認する。
10. タブ切り替え、POST後のエラー表示、エラーがあるタブへの自動切り替えを確認する。

## 11. 変更してよいもの / 変更しないもの

変更してよいもの:

- Razor上のHTML構造
- CSSクラス
- 余白、色、角丸、影
- ラベル、補足文、見出し文言
- エラー表示位置
- レスポンシブ配置

UI調整だけでは変更しないもの:

- ViewModelのプロパティ構造
- Validatorのルール
- Mapper / Normalizerの処理
- Domain Entity
- 計算ロジック
- POSTされる入力項目の意味

## 12. 確認観点

- Figmaの主要な見た目が再現されている。
- パディング、マージン、フォントサイズ、角丸、影、色が共通定義に寄せられている。
- 同じ見た目のCSSやHTML構造が不要に重複していない。
- 既存の入力値保持が壊れていない。
- バリデーションエラーが入力項目の近くに表示される。
- POST後に入力値が保持され、エラーがあるタブや入力項目を見つけやすい。
- タブ、ツールチップ、ポップオーバーなど状態を持つUIのARIA属性が役割と一致している。
- スマホ幅で横スクロールが発生しない。
- ボタン、タブ、入力欄の操作対象が小さすぎない。
- `dotnet build` が成功する。
- 必要に応じてブラウザで主要画面を確認する。
